1. Vision – What NDRA‑PII Does

NDRA‑PII is an enterprise‑grade system that ingests unstructured content (documents, emails, scans, images, voice transcripts), detects and classifies PII, understands context, scores risk, applies policy rules, redacts or quarantines, and produces fully explainable, auditable decisions.[web:1][web:2]

Microsoft Presidio provides deterministic, high‑precision PII detection and anonymization, while NDRA supplies the multi‑agent orchestration, semantic retrieval, rule logic (NSRL), UI, risk analytics, and multi‑tenant governance that wrap around it.[web:1][web:6]

2. High‑Level Architecture (with Presidio)

NDRA‑PII is structured as a layered, multi‑tenant platform: ingestion, document intelligence, Presidio‑based PII detection, retrieval and vector store, multi‑agent swarm, decision engine, and UI/audit layers.[web:1][web:5]

Presidio's AnalyzerEngine and AnonymizerEngine sit in the PII Detection and Remediation layers, while NDRA's agents handle OCR, chunking, retrieval, reasoning, risk, explainability, and governance across tenants.[web:2][web:6]

Functional Layers

  • Ingestion & Auth (UI, API, Email, SDK)
  • Document Intelligence (OCR, layout, chunking)
  • Presidio PII Detection (AnalyzerEngine + custom recognizers)
  • Retrieval & Vector Store (dynamic embeddings)
  • Multi‑Agent Orchestration (PII, NSRL, Risk, Redaction)
  • Decision Engine & Actions
  • UI, Dashboards, Audit & Governance

Presidio's Role

  • Context‑aware PII detection with predefined/custom recognizers.[web:1]
  • Anonymization operators (mask, redact, replace, hash, tokenize).[web:2][web:15]
  • Text, image, and structured data support through dedicated modules.[web:2]
  • Extensible and deployable via Python, containers, or Kubernetes.[web:1][web:2]

3. Core Pipeline – NDRA‑PII + Presidio

Step 1 – Ingest Document

Documents are uploaded via web UI, APIs, email harvester, or SDKs and stored in an object store with a corresponding document record in Postgres that captures tenant, file type, and status metadata.[web:5][web:6]

An asynchronous job is queued for preprocessing, allowing the system to scale horizontally and decouple ingestion from heavy processing.[web:5]

Step 2 – Document Intelligence

OCR engines (e.g., Tesseract or PaddleOCR) and layout models (e.g., LayoutLM/Donut) extract text, detect blocks/tables/forms, and map tokens to page coordinates.[web:6][web:7]

A semantic chunker creates clause‑aware chunks (roughly 300–800 tokens) with metadata such as page, section, heading, and token count, enabling downstream retrieval and explainability.

Step 3 – Presidio PII Extraction

For each chunk, Presidio's AnalyzerEngine detects PII entities using a mix of NER, regular expressions, rule‑based logic, and checksum validation, with support for multilingual detection and custom recognizers.[web:1][web:2]

Each detection yields entity type, score, start/end positions, and recognizer details, which NDRA normalizes into PII Objects attached to chunk IDs and PDF coordinates for later highlighting.[web:6]

Step 4 – PII Fusion Agent

A PII Fusion Agent merges Presidio results with NER and LLM‑based evidence extraction, resolving overlaps and conflicts into unified PII Objects that track detection method and confidence.[web:1][web:10]

This hybrid approach lets the system handle long‑tail or domain‑specific entities while still relying on Presidio's deterministic core for standard PII pattern coverage.[web:6][web:8]

Step 5 – Retrieval Layer (Optional)

An Embedding Agent routes between models such as MiniLM for generic text, LegalBERT for legal contracts, and BioBERT for healthcare forms, then populates a distributed vector DB like Milvus, Pinecone, or Chroma.[web:5][web:8]

A Retrieval Agent performs hybrid BM25 + dense retrieval to surface the most relevant clauses and context around PII when reasoning about risk and policy compliance.

Step 6 – NSRL Reasoning Engine

The Neuro‑Symbolic Rule Language (NSRL) expresses data‑handling rules over PII types, document types, and contextual metadata (e.g., channel, jurisdiction, tenant policy) as declarative rules.[web:10]

An NSRL Reasoner Agent compiles these rules into executable logic, evaluates them against PII Objects and document metadata, and uses LLMs for ambiguous clauses or natural‑language policy interpretation.

Step 7 – Risk Scoring Agent

A Risk Analyzer Agent computes a risk_score (0–100) based on the combination of PII categories, counts, density, exposure channel, and document type, informed by research on adaptive PII mitigation frameworks.[web:10]

It produces risk_flags (e.g., "identity cluster", "financial + contact"), recommended_actions, and inputs to dashboards and triage views.

Step 8 – Presidio Redaction Layer

When rules and risk outcomes require mitigation, a Redaction Agent uses Presidio's AnonymizerEngine to mask, remove, replace, hash, or tokenize detected entities in text, driven by operator configuration per PII type.[web:2][web:15]

For PDFs and images, NDRA maps spans to coordinates and applies visual redaction (e.g., black boxes, blur), using Presidio's image redaction capabilities where possible and maintaining an internal redaction map.[web:2]

Step 9 – Explainability & Audit

An Explainability Agent builds a decision timeline that shows which agents ran, which chunks and spans were used, which rules fired, and any LLM reasoning snippets used in borderline decisions.[web:12]

An audit log captures inputs, outputs, rule sets, and hashes of decision payloads, which can be forwarded to SIEM tools and used for compliance evidence and forensic investigations.[web:12][web:20]

Step 10 – UI Delivery

A web UI displays PDF pages with PII highlights, risk scores, action summaries, rule explanations, and side‑by‑side original vs. redacted previews to help reviewers understand and approve decisions.[web:5]

Triaging users can filter by tenant, channel, risk level, or document type and export decision JSON for integration with downstream systems or case management tools.

4. NDRA‑PII Agents (Full Swarm)

NDRA‑PII runs as a multi‑agent swarm where each agent implements a focused responsibility and communicates via topics/queues, enabling horizontal scaling, resilience, and clear boundaries for testing.[web:13][web:16]

Presidio integration is concentrated in the PII Extractor and Redaction Agents, while other agents focus on OCR, retrieval, rules, risk, and explainability around those PII primitives.[web:1][web:2]

5. PII Types & Presidio Capabilities

Presidio includes predefined recognizers for common PII such as credit cards, SSNs, phone numbers, emails, IP addresses, names, and locations, and can be extended with custom recognizers for regional and domain‑specific IDs.[web:1][web:2]

NDRA‑PII organizes these into categories (government IDs, contact info, financial data, personal data, sensitive health/biometric references) and maps them onto NSRL rule conditions and risk scoring logic.[web:8][web:10]

Out‑of‑the‑Box

  • Government IDs (SSN, tax IDs, national IDs).[web:1]
  • Contact data (emails, phone numbers, IPs, URLs).[web:2]
  • Financial data (credit/debit cards, bank details).[web:1][web:6]
  • Names, locations, addresses, dates.[web:2]
  • Custom NER‑based entities via spaCy/Transformers hooks.[web:1]

Anonymization Operators

  • Masking (e.g., "XXXX‑XXXX‑1234").[web:15]
  • Redaction ("<REDACTED>" or style‑based removal).[web:15]
  • Replacement and pseudonymization tokens.[web:6][web:15]
  • Hashing (SHA/HMAC) and reversible tokenization.[web:18]

6. Data Models & NSRL DSL

NDRA‑PII standardizes the flow with Document, Chunk, PII Object, and Decision JSON models, which agents read and write across the pipeline.[web:5][web:6]

The Neuro‑Symbolic Rule Language (NSRL) defines data‑handling rules as YAML/JSON conditions over these objects, compiled into executable rule logic by a dedicated Rule Engine.[web:10]

Core Objects (Conceptual)

  • Document: IDs, tenant, filename, pages, MIME type, storage URL, status.
  • Chunk: chunk_id, doc_id, page, text, coords, embedding_vector, metadata.
  • PII Object: pii_id, doc_id, chunk_id, type, subtype, text, method[], confidence, coords.
  • Decision: decision_id, doc_id, pii[], risk_score, rules_fired[], suggested_action, explainability, audit_hash.

NSRL Rule Examples (Conceptual)

Rules are expressed as high‑level conditions (e.g., "Aadhaar in contract → high severity") that compile into Python rule functions and can be versioned, tested, and overridden as policy changes.[web:10]

Conditions can match PII subtype, combinations (e.g., ID + address + DOB), document type, source channel, and jurisdiction, enabling fine‑grained governance.

7. Difficulty & Build Timeline (Presidio‑Assisted)

Using Presidio significantly reduces the difficulty of NDRA‑PII because it removes the need to handcraft regexes, checksums, and anonymizers for a broad set of PII types.[web:1][web:8]

This lets the team concentrate on orchestration, retrieval, rules, risk, and UI, which are more differentiated and higher‑value than low‑level pattern work.[web:6][web:12]

8. Security, Compliance & KPIs

Presidio offers core capabilities for de‑identification and redaction but warns that automated detection cannot guarantee discovery of all sensitive data, so it should be combined with additional protections and governance.[web:2]

NDRA‑PII adds multi‑tenant isolation, SSO/OAuth2, RBAC, encrypted storage, SIEM forwarding, and full decision logging so that deployments can align with privacy regulations and internal security policies.[web:12][web:20]

9. Recommended Tech Stack

The platform is intentionally built from widely used, open‑source‑friendly components so it can be prototyped quickly and hardened for enterprise later.[web:1][web:5]

Presidio is consumed as a Python SDK or container service; all other components use standard backend, frontend, and data infrastructure technologies.[web:1][web:2]

Backend & Infra

  • FastAPI (Python) for APIs.
  • Microsoft Presidio Analyzer & Anonymizer.[web:1][web:2]
  • Postgres for relational data.
  • Redis for caching.
  • Kafka (or similar) for events.
  • Docker + Kubernetes for deployment.

AI, Retrieval & UI

  • Presidio recognizers + custom NER models.[web:1]
  • LayoutLM/Donut for layout‑aware OCR.[web:7]
  • Sentence Transformers for embeddings.
  • Llama/Gemini (or equivalents) for reasoning.[web:10]
  • Milvus / Pinecone / Chroma as vector DB.[web:5][web:8]
  • Next.js + TailwindCSS + PDF.js for UI.[web:5]

10. Example End‑to‑End Workflow

A typical flow: a user uploads a contract, NDRA‑PII chunks and runs it through Presidio, NSRL rules, risk scoring, and redaction, then surfaces a fully explainable decision in the UI with download and API hooks.[web:5][web:6]

This pattern generalizes to emails, scanned forms, and other unstructured sources, with only the ingestion adapters and OCR parameters changing.

  1. User uploads a document via web or API.
  2. Preprocessing Agent runs OCR, layout, and chunking.
  3. PII Extractor Agent runs Presidio AnalyzerEngine on each chunk.[web:1][web:2]
  4. PII Fusion Agent merges Presidio, NER, and LLM spans.
  5. NSRL Reasoner evaluates policy rules on PII and context.[web:10]
  6. Risk Analyzer computes risk_score and recommended actions.[web:10]
  7. Redaction Agent uses Presidio AnonymizerEngine to redact.[web:2][web:15]
  8. Explainability Agent builds timeline and highlights.
  9. UI presents risk, actions, and previews; audit entry is logged.[web:12]
