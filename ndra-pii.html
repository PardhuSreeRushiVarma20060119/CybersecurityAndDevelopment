NDRA-PII
Neuro-Semantic Multi-Agent PII Intelligence System
Technical Design Document (with Microsoft Presidio)

This document describes the full engineering design of NDRA-PII: a neuro-semantic, multi-agent PII intelligence system integrating Microsoft Presidio for detection and anonymization.

1. System Overview

NDRA-PII is a multi-agent platform for PII detection, contextual reasoning, rule-based compliance enforcement, redaction, and auditability.

  • Supports documents, scans, images, emails, OCR text
  • Uses Microsoft Presidio for deterministic PII detection
  • Uses LLMs for semantic reasoning & contextual inference
  • Uses a Neuro-Symbolic Rule Layer (NSRL) for policy logic

2. Architecture

2.1 Layered Architecture

  • Ingestion Layer: Upload, email intake, API intake
  • Document Intelligence: OCR, layout detection, semantic chunking
  • PII Detection Layer: Presidio AnalyzerEngine + custom NER
  • Semantic Retrieval Layer: Embedding computation, vector DB, hybrid retrieval
  • Agent Orchestration: Kafka or event-driven microservices
  • Reasoning Layer: NSRL rule interpreter
  • Risk Layer: Quantified scoring model
  • Redaction Layer: Presidio Anonymization Engine
  • Explainability + Audit: Provenance, rule traces, decision JSON
  • UI Layer: PDF viewer, highlights, dashboards

3. End-to-End Pipeline

3.1 Step 1 — Ingestion

Accept files via API or UI. Store in object storage. Queue processing.

3.2 Step 2 — OCR & Chunking

Use PaddleOCR or Tesseract → text extraction. Use LayoutLM/Donut → tables/forms segmentation. Chunk text into 300–800 token semantic segments.

3.3 Step 3 — Presidio PII Detection

Run AnalyzerEngine per chunk: 

entity_type, start, end, score, recognizer_type

3.4 Step 4 — PII Fusion

Merge:

  • Presidio recognitions
  • NER outputs
  • LLM-based evidence

3.5 Step 5 — Retrieval

Compute embeddings (MiniLM, LegalBERT). Store in vector DB (Milvus/Pinecone/Chroma). Retrieve context for reasoning.

3.6 Step 6 — NSRL Reasoning

Apply organizational policies.

Example: 
WHEN pii.subtype == "aadhaar"
AND doc.type == "contract"
THEN action = escalate
THEN severity = high

3.7 Step 7 — Risk Scoring

Compute risk_score based on PII quantity, types, document channel, exposure.

3.8 Step 8 — Redaction

Use Presidio AnonymizerEngine:

  • mask
  • replace
  • tokenize
  • remove

3.9 Step 9 — Explainability

Produce:

  • chunk usage
  • PII span maps
  • rule firing order
  • audit hash

3.10 Step 10 — Output Delivery

UI displays risk ratings, PII highlights, redacted documents.

4. Multi-Agent Architecture

Agent Description Uses Presidio?
Preprocessing Agent OCR + layout analysis + chunking No
PII Extractor Agent Runs AnalyzerEngine on each chunk Yes
Fusion Agent Join Presidio, NER, LLM spans Indirect
Retrieval Agent Hybrid semantic search No
NSRL Reasoner Execute rule DSL Consumes Presidio outputs
Risk Agent Compute risk_score Consumes Presidio outputs
Redaction Agent Uses AnonymizerEngine for masking Yes
Explainability Agent Event timeline + audit No

6. Rule Language (NSRL)

6.1 DSL Format

rule_id: R_HIGH_AADHAAR
when:
  - pii.subtype == "aadhaar"
  - doc.type == "contract"
then:
  - severity: high
  - action: escalate

6.2 Engine Requirements

  • YAML parser → AST
  • AST → Executable rule functions (Python)
  • Support priorities & override rules
  • Version-controlled rulesets

7. System KPIs

  • PII Precision ≥ 95%
  • PII Recall ≥ 92%
  • False Positive Rate ≤ 5%
  • Latency per document ≤ 8s
  • Redaction Accuracy ≈ 99%
  • Audit Trace Coverage: 100%

9. Example Workflow

  1. User uploads file
  2. OCR + chunking
  3. Presidio identifies PII
  4. Fusion agent consolidates spans
  5. NSRL evaluates rules
  6. Risk agent computes score
  7. Presidio redacts
  8. Explainability agent logs decision
  9. UI displays output
